# 软件开发 Loop 工程 · 工程详解

> 这份文档是给**想看懂这套循环怎么转、为什么这么设计、每个文件干什么**的人读的。
> 只想快速上手用，看根目录 `README.md` 就够；想改造/扩展/排错，再读这份。
>
> 内容分两大块：**前半（1–11 节）**讲设计与运转原理，**后半（第 12 节）**逐文件解释整个工程目录。

---

## 1. 这是什么

一条**自动化软件开发循环**的通用脚手架，跑在 Claude Code 上。
你把需求喂进去，它**自己**走一圈：发现任务 → 隔离开发 → 写代码 → 独立验收 → 合并/开 PR → 记录进度，然后下一圈接着干。

设计依据《Loop Engineering 橙皮书》（作者 alchaincyf）。核心思想一句话：

> **让"创造"和"把关"由两套不同的东西负责——会犯错的模型负责写，确定性的脚本负责守底线。**

为什么要分开？因为让写代码的 AI 给自己打分，它总会觉得"我做完了"。所以这套循环里：
- **创造层**（会犯错、但有创造力）：skill 和子 agent，负责发现、实现、评判。
- **确定性 gate 层**（不会犯错、但只会按规则判断）：hook（bash 脚本），负责守预算、拦危险命令、卡"全绿才放行"的硬门。

能写死成规则的判断，绝不交给模型——这是整套设计的主轴。

---

## 2. 橙皮书：五动作 ↔ 六零件

| 动作 | 在干什么 | 对应零件 | 本工程的落点 |
|------|----------|----------|--------------|
| **发现** Discovery | 找出这一圈值得做的事 | Skills | `loop-triage` |
| **交付** Handoff | 把一个任务隔离出来开发 | Worktrees | `loop-implement`（`git worktree`） |
| **验证** Verification | 独立把关，能说"不" | Sub-agents | `loop-review` + `generator`/`evaluator` + `gate-stop.sh` |
| **持久化** Persistence | 把成果落到对话之外 | Memory + Connectors | `loop-persist` + `memory/*` + MCP |
| **调度** Scheduling | 安排下一圈 | Automations | `loop.md` + `/loop` 间隔运行器 |

六零件 = Skills / Worktrees / Sub-agents / Memory / Connectors / Automations。
五动作是"做什么"，六零件是"用什么做"。

## 3. 整体架构（一图看懂分层）

```
                 ┌─────────────────── /loop-cycle（一圈入口） ───────────────────┐
                 │            按 .claude/loop.md 顺序编排，严格不跳步             │
                 └───────────────────────────────┬──────────────────────────────┘
                                                 │
   ┌─────────────────────────── 创造层（会犯错，但有创造力）───────────────────────────┐
   │                                             │                                       │
   │  发现        交付              验证                          持久化                 │
   │  loop-triage loop-implement    loop-review                  loop-persist           │
   │     │           │                 │                            │                   │
   │     │       开 worktree      派 evaluator 子 agent         开 PR / 合并             │
   │     │       派 generator     （怀疑论、会动手验证）         更新状态文件             │
   │     ▼           ▼                 ▼                            ▼                   │
   └─────┼───────────┼─────────────────┼────────────────────────────┼───────────────────┘
         │           │                 │                            │
   ┌─────┴───────────┴─────────────────┴────────────────────────────┴───────────────────┐
   │              确定性 gate 层（不会犯错，只按规则判断；LLM 跳不过）                    │
   │   token-guard.sh（每次工具调用前守预算）                                            │
   │   danger-guard.sh（每条 Bash 命令前拦危险动作）                                     │
   │   gate-stop.sh（收尾时卡 test/lint/build 全绿）                                     │
   └────────────────────────────────────────────────────────────────────────────────────┘
                                                 │
   ┌─────────────────────────────────────────────┴──────────────────────────────────────┐
   │   记忆（对话之外的状态）：loop-state.md（待办/进行中/已完成/游标）                  │
   │                          inbox.md（循环搞不定的事，留给人）                          │
   │                          budget.json（预算计数）                                     │
   │   配置：loop.env（唯一配置文件）   连接器：.mcp.json（Playwright 等 MCP）           │
   └─────────────────────────────────────────────────────────────────────────────────────┘
```

记住三件事就抓住了全貌：
1. **上层会犯错、下层不会**——所以危险的事全压在下层 hook。
2. **生成者和评判者是两个不同的子 agent**——绝不自己给自己打分。
3. **所有命令/参数只在 `loop.env` 一个文件里**——套用新项目只改它。

## 4. 一圈完整流程（逐步拆解）

入口是 `/loop-cycle` 命令（`.claude/commands/loop-cycle.md`），它读 `.claude/loop.md` 并严格按步骤执行。

### 第 0 步 · 急停 + 重置单圈计数
- 若 `.claude/memory/STOP` 文件存在 → 整圈立即跳过（最快的刹车）。
- 否则把 `budget.json` 的 `loop_calls` 归零（单圈预算重新计）。

### 第 1 步 · 发现（`loop-triage`）
从四个来源找活，**优先级最高的是你的需求文档**：
1. **需求文档**：项目根 `需求文档.md` / `需求/*.md` / `BACKLOG.md`——新老项目喂活的主入口。
2. CI 失败（修红）。
3. open issue（需接 GitHub MCP）。
4. 代码里的 `TODO`/`FIXME`/`HACK`。

triage 把值得做的拆成**边界清晰、能在一个 worktree 里独立完成**的任务，写进 `loop-state.md` 的「待办」；模糊、需要人判断的写进 `inbox.md`。

> **新项目 vs 老项目的关键差别就在这一步**：
> - 新项目（仓库几乎没源码）→ 直接按需求拆任务。
> - 老项目（已有成体量源码）→ 先用 Glob/Grep **读懂需求关联的现有代码**，再拆任务，并在任务里**标注受影响文件/函数**，让后面的 generator 能最小改动、evaluator 知道要回归哪些存量行为。

每个任务还会按 `PROJECT_MODE` 打层标签 `[fe]`/`[be]`/`[both]`，决定后续进哪个目录、用哪种验证。

待办为空就结束本圈（不空转）。

### 第 2 步 · 取任务
从「待办」取**优先级最高的 1 个**，移到「进行中」。**一圈只做一个**，控制影响面。

### 第 3 步 · 交付 + 生成（`loop-implement` → `generator`）
1. 用原生 `git worktree` 从最新主分支切出隔离目录 `.worktrees/loop-<任务短名>`（仓库内、已 gitignore）、分支 `loop/<任务短名>`，互不踩脚。放仓库内而非同级 `../`，是为了让 worktree 留在工作区根之下，全自动跑循环时 Claude Code 不会弹"访问工作区外目录"的确认框。
2. 按层标签定位目录与命令（`FE_*` / `BE_*`）。
3. 派 `generator` 子 agent 实现：
   - **老项目先读关联现有代码**再动手，最小改动、不破坏存量。
   - 实现后**必须自测**（跑对应侧 test/lint）。
   - generator **不给自己打分**——好坏由下一步独立评判者说了算。

### 第 4 步 · 验证（`loop-review` → `evaluator`）
派一个**和 generator 完全不同指令、可换更强模型**的 `evaluator` 子 agent：
- 默认立场：**这段代码是坏的，除非被证明能跑**。
- **必须动手**，按层切换验证方式：
  - `[fe]` → 跑测试 + 用 Playwright MCP 开真浏览器点页面、截图、查 DOM。
  - `[be]` → 跑测试 + 真发 HTTP 请求验接口状态码/响应体/落库。
- 对照**完成标准**逐条核；老项目还要查**回归**（旧功能没被改坏）。
- 裁决：通过 → 进持久化；不通过 → 退回 generator 重修，最多 `MAX_FIX_ATTEMPTS` 次；仍不过 → **不推进**，写 `inbox.md` 转人工。

### 第 5 步 · 持久化（`loop-persist`）
1. 推分支、开 PR（写清做了什么/为什么/验证结果）。
2. **合并前置条件（缺一不可）**：evaluator 通过 **且** `gate-stop.sh` 全绿 **且** 本任务那一侧命令在 loop.env 里真填了（不是占位符）。
   - **防零验证合并硬卡**：若该侧命令仍是占位符 `<...>` 或空，说明 gate-stop 当时是"跳过该侧"而非真跑了测试 → **绝不自动合并**，写 inbox 转人工。"未配置 ≠ 通过"。
3. 按档位决定：
   - **C 档**（`AUTO_MERGE=true`）→ 自动合并到主分支。
   - **B 档**（`AUTO_MERGE=false`）→ 只留 PR 等人 review+merge。
4. 更新 `loop-state.md`（任务移到「已完成」），清理用完的 worktree。

### 第 6 步 · 调度游标
更新 `loop-state.md` 的「游标」（本圈到哪、下圈从哪继续），结束本圈。下次 `/loop` 自动接上。

**全程**：每次工具调用前 `token-guard` 守预算、每条 Bash 命令前 `danger-guard` 拦危险动作。

## 5. 三道护栏（hook）详解

护栏是这套循环敢"自动合并"的底气。它们是确定性 bash 脚本，**LLM 跳不过**。注册在 `.claude/settings.json`。

### 5.1 `danger-guard.sh`（PreToolUse，matcher=Bash）
每条要执行的 Bash 命令前拦一道。命中即 **exit 2 阻断**：
- `rm -rf`/`rm -fr`、fork 炸弹、`mkfs`/`dd if=` 磁盘破坏
- `git push --force` / `--force-with-lease` 强推
- `git reset --hard`、`git clean -f/-d`、`git branch -D`、`git checkout .`/`git restore .`（丢改动）
- `--no-verify`（跳校验钩子）
- **直接 push 到主分支**（即使非 force；C 档由循环自动合并，人/agent 不该手动直推 main）

> 有 jq 时精确解析命令文本；无 jq 时退化为整段 JSON 匹配（仍能拦，但更粗）。

### 5.2 `token-guard.sh`（PreToolUse，matcher=*）
每次工具调用前计数，超 `PER_LOOP_BUDGET`（单圈）或 `DAILY_BUDGET`（每日）即 exit 2，**防空转 bug 烧光额度**。计数落在 `memory/budget.json`，跨天自动重置。
- 预算是占位符/空 → 跳过守卫并提醒。
- **无 jq 时的折中**：jq 是计数硬前提，但不能把 agent 彻底锁死（连写 inbox 转人工都做不到）。所以**只 fail-closed 卡住烧钱主驱动 Bash**，其余工具放行但大声告警——让人看见护栏没生效，赶紧装 jq。

### 5.3 `gate-stop.sh`（Stop hook）
收尾时按 `PROJECT_MODE` 跑对应侧的 **test → lint → build**，**全绿才 exit 0 放行**，任一红 exit 2 让模型继续修。
- 命令是占位符/空 → 跳过该侧并警告"未配置=未验证"（真正的硬卡在 loop-persist：未配置侧禁止自动合并）。
- 这是合并前**最后一道确定性闸**，比评判者更不讲情面。

## 6. 唯一配置文件 `loop.env`

套用到新项目**只改这一个文件**。格式是可被 `bash source` 的 `KEY="value"`。

| 键 | 作用 |
|----|------|
| `PROJECT_MODE` | `frontend`(纯前端) / `backend`(纯后端) / `fullstack`(前后端)——决定填哪组、验哪侧 |
| `FE_LANG` / `FE_DIR` / `FE_TEST_CMD` / `FE_LINT_CMD` / `FE_BUILD_CMD` / `FE_RUN_CMD` | 前端语言/目录/命令（含前端时填） |
| `BE_LANG` / `BE_DIR` / `BE_TEST_CMD` / `BE_LINT_CMD` / `BE_BUILD_CMD` / `BE_RUN_CMD` | 后端语言/目录/命令（含后端时填） |
| `MAIN_BRANCH` | 主分支名（默认 main） |
| `AUTO_MERGE` | `true`=C 档自动合并 / `false`=B 档只开 PR（切档位只改这一个值） |
| `MAX_FIX_ATTEMPTS` | 评判打回后 generator 最多重修几次（建议 3） |
| `PER_LOOP_BUDGET` / `DAILY_BUDGET` | 工具调用预算上限 |
| `MAX_RETRIES` | 预留，暂未启用 |
| `MCP_CONFIG` | 留空=不接外部系统；接 GitHub/issue/Slack 时在 MCP 配 |

> `*_RUN_CMD` 是给评判器起服务用的（前端起浏览器验、后端起服务发请求）。
> 项目规约（命名/分层/测试约定）写在 `CLAUDE.md` 的「项目规约」一节，**不放 loop.env**。

## 7. C 档 / B 档：自主程度的旋钮

| 档 | `AUTO_MERGE` | 行为 | 适合 |
|----|--------------|------|------|
| **C 档** | `true` | 评判通过 + 全绿 → **自动合并 main** | 信任建立后、可承受风险的场景 |
| **B 档** | `false` | 评判通过 + 全绿 → **只开 PR，停下等人 review+merge** | 生产环境（推荐） |

切换只改 `loop.env` 一个值。**人没退场，只是把时间从"写"挪到"审"。**

橙皮书建议的渐进路径（§09）：
1. 先只跑 `loop-triage`（只发现、不改代码），跑顺。
2. 接上 implement+review，但停在 B 档（开 PR 不合并）。
3. 信任建立、确认三道护栏都生效后，再开 C 档。

## 8. 记忆（对话之外的状态）

| 文件 | 作用 |
|------|------|
| `memory/loop-state.md` | 状态主文件：待办 / 进行中 / 已完成 / 游标 |
| `memory/inbox.md` | 收件箱：循环搞不定、验证多次不过、或触红线的事，留给人 |
| `memory/budget.json` | 预算计数（token-guard 读写） |
| `memory/STOP` | 急停文件（存在即整圈跳过；删掉恢复） |

记忆让循环**跨圈、跨会话**保持连续——下一圈知道上一圈干到哪。

## 9. 套用到任意项目（新老都兼容）

```bash
bash install-loop.sh <目标项目目录>
```

脚本会：装强制依赖 jq/gh（装不上则中止，不允许无护栏运行）→ 拷 `.claude/`、`CLAUDE.md`、`.gitattributes`、`.mcp.json` → 重置运行态记忆。

**对老项目的保护（绝不覆盖你已有的配置）**：
- 已有 `.claude` → 自动备份成 `.claude.bak.<时间戳>` 再装新的。
- 已有 `CLAUDE.md` → 循环总纲放成 `CLAUDE.loop.md`，**你手动把其中「安全约定」并入你的 CLAUDE.md**。
- 已有 `.mcp.json` → 循环 MCP 放成 `.mcp.loop.json`，**含前端时把 playwright 服务器并入你的 .mcp.json**。

这套设计是**幂等安全**的：重复装、装进有东西的老仓库，都只新增/备份，绝不破坏。

**目标项目要求**：必须是 git 仓库、有主分支、已连 GitHub 远程（worktree 隔离 + `gh pr create` 开 PR 都依赖它）。

## 10. 安全约定与三种急停

**强制安全约定**（`CLAUDE.md`）：
1. 一圈只推进一个任务。
2. 生成者与评判者必须是不同子 agent。
3. 任何 gate 不过 → 不推进，写 inbox 等人。绝不绕硬门、绝不 `--no-verify`、绝不删测试"过关"。
4. 禁止 `push --force` 主分支、`reset --hard`、`rm -rf`、删分支、改 CI——由 `danger-guard` 拦。
5. 能写死规则的判断交给 hook，不交给模型。

**三种急停**：
1. `export CLAUDE_CODE_DISABLE_CRON=1` —— 关所有定时。
2. `touch .claude/memory/STOP` —— 每圈开头检测到即整圈跳过（最快）。
3. 直接停 `/loop` 会话。

## 11. 常见坑（FAQ）

**Q：Windows 上 hook 不生效？**
A：`settings.json` 用 `bash .claude/hooks/*.sh` 调起，要求 git-bash 的 bash 在 PATH。`.sh` 必须是 LF 换行（`.gitattributes` 已强制；CRLF 会让 bash 报 `\r` 错）。冒烟验证：让 agent 试跑一条 `rm -rf` 看是否被拦（退出码 2）。

**Q：install-loop.sh 自动装 jq/gh 失败？**
A：Windows git-bash 调 winget 常报 `Permission denied`。改在真实终端（PowerShell/CMD）手动装：`scoop install jq gh` / `choco install jq gh -y` / `winget install jqlang.jq` + `winget install GitHub.cli`，装好 `gh auth login` 一次。

**Q：缺 jq 会怎样？**
A：预算守卫无法计数 → 退化为只拦烧钱主驱动 Bash、其余放行并大声告警。**务必先装 jq 再跑循环**。

**Q：前端浏览器评判用不了？**
A：靠项目根 `.mcp.json` 的 Playwright MCP（`npx @playwright/mcp` 首次自动下载，需 Node/npx）。首次加载 `.mcp.json` 会弹 MCP 批准框，点同意。

**Q：循环卡住不推进？**
A：八成是某个 gate 没过——看 `inbox.md`，里面写了为什么转人工。

## 12. 工程目录详解（逐文件）

先看整棵树，再逐个解释。带 ★ 的是你套用到新项目时**会去改**的；其余是引擎，通常不用动。

```
loop工程/
├── README.md                      使用入口：简介 + 教程 + 说明
├── LICENSE                        MIT 许可证
├── CLAUDE.md                  ★   总纲：配置指引 / 项目规约 / 安全约定 / 急停（模型每次都读）
├── install-loop.sh                一键安装脚本：把模板装进任意项目
├── .gitattributes                 强制 .sh 用 LF 换行（防 Windows CRLF 把 bash 搞挂）
├── .gitignore                     忽略 *.pdf、本地私有 .mcp.local.json
├── .mcp.json                      项目级 MCP：自带 Playwright，clone 即用（前端评判靠它）
├── .mcp.json.example              加 GitHub/数据库等连接器的参考模板（token 用 ${ENV_VAR}）
├── doc/
│   └── 工程详解.md                就是本文件
└── .claude/                       Claude Code 的工程配置根目录（引擎全在这）
    ├── loop.env               ★   唯一配置文件：模式 / 语言 / 命令 / 档位 / 预算
    ├── loop.md                    一圈六步的编排剧本（由 /loop-cycle 执行）
    ├── settings.json              注册 3 个 hook（把护栏挂到 Claude Code 事件上）
    ├── commands/
    │   └── loop-cycle.md          /loop-cycle 斜杠命令：跑一圈的入口
    ├── skills/                    四个技能 = 一圈的四个主动作
    │   ├── loop-triage/SKILL.md   发现：读需求/CI/issue/TODO，拆任务进待办
    │   ├── loop-implement/SKILL.md 交付：开 worktree、派 generator
    │   ├── loop-review/SKILL.md   验证：派 evaluator 独立把关
    │   └── loop-persist/SKILL.md  持久化：开 PR / 合并 / 更新状态
    ├── agents/                    两个子 agent = 写的人 和 把关的人（必须不同）
    │   ├── generator.md           生成者：只负责实现 + 自测，不给自己打分
    │   └── evaluator.md           评判者：默认怀疑、会动手验证、可换更强模型
    ├── hooks/                     三道确定性护栏（bash 脚本，LLM 跳不过）
    │   ├── gate-stop.sh           全绿硬门：收尾跑 test/lint/build，全绿才放行
    │   ├── token-guard.sh         预算守卫：每次工具调用前计数，超额即停
    │   └── danger-guard.sh        危险拦截：rm -rf / 强推 / 直推 main 等一律拦
    └── memory/                    对话之外的状态（让循环跨圈跨会话连续）
        ├── loop-state.md          状态主文件：待办 / 进行中 / 已完成 / 游标
        ├── inbox.md               收件箱：循环搞不定的事，留给人
        ├── budget.json            预算计数（token-guard 读写，跨天重置）
        └── STOP                   急停文件（默认没有；touch 出来即整圈跳过）
```

### 12.1 根目录文件

| 文件 | 干什么 | 你要不要改 |
|------|--------|-----------|
| `README.md` | 使用入口：通俗简介 + 上手教程 + 使用说明 | 看，不改 |
| `CLAUDE.md` | **总纲**。Claude Code 每次都会读它，里面写了配置指引、**项目规约**（命名/分层/测试约定，给模型读的自由文本）、强制安全约定、三种急停 | ★ 套用后在「项目规约」一节写本项目约定 |
| `install-loop.sh` | 一键把整套模板装进目标项目；强制装 jq/gh；老项目不覆盖你已有配置（放 sidecar） | 不改，直接跑 |
| `.gitattributes` | 强制 `*.sh` 为 LF 换行。Windows 上若变 CRLF，bash 会报 `\r` 错、hook 全挂 | 不改 |
| `.gitignore` | 忽略 PDF 和本地私有 MCP 覆盖文件 | 按需 |
| `.mcp.json` | 项目级 MCP 配置，**已自带 Playwright**——别人 clone 就能用浏览器做前端评判，不需要任何全局配置 | 要加 GitHub/DB 连接器时改 |
| `.mcp.json.example` | 加连接器的参考写法，强调 token 用 `${ENV_VAR}` 不硬编码 | 参考 |
| `LICENSE` | MIT，可自由用/改/商用 | 不改 |
| `doc/工程详解.md` | 本文件 | 看 |

### 12.2 `.claude/`：引擎所在

这是 Claude Code 读取工程配置的固定目录。整条循环的"大脑和手脚"都在这里。

**配置与编排**
- `loop.env` ★ —— **唯一配置文件**。套用新项目你基本只动它（详见第 6 节）。其它文件都从这里读命令/参数，所以换项目不用改它们。
- `loop.md` —— 一圈六步的剧本（第 0–6 步，见第 4 节）。`/loop-cycle` 就是照它执行。
- `settings.json` —— 把三个 hook 注册到 Claude Code 的事件上（`Stop` 挂 gate-stop；`PreToolUse` 挂 danger-guard 和 token-guard）。Windows 必须用 `bash .claude/hooks/*.sh` 调起。
- `commands/loop-cycle.md` —— 定义 `/loop-cycle` 这个斜杠命令，是"跑一圈"的人机入口。

**skills/（四个主动作）** —— 每个技能是一段给模型的指令，对应一圈里的一个动作；它们只下达"做什么"，具体命令都回到 `loop.env` 取。
- `loop-triage` 发现 · `loop-implement` 交付 · `loop-review` 验证 · `loop-persist` 持久化（细节见第 4 节）。

**agents/（两个角色）** —— 子 agent 是带独立上下文的"分身"。这里**刻意分成两个**：
- `generator.md` 写代码的人——只实现、只自测，倾向于觉得"我做完了"。
- `evaluator.md` 把关的人——默认"代码是坏的除非证明能跑"，会真跑命令/真开浏览器，**可换更强模型**。
- 为什么分两个？让写代码的给自己打分太手软，所以验收必须是另一个角色。

**hooks/（三道护栏）** —— 确定性 bash 脚本，是循环敢自动化的底气（详见第 5 节）。和 skill/agent 的区别：**hook 不会犯错也没有创造力，只按写死的规则判断**，所以危险的、不能商量的判断全压给它们。

**memory/（状态）** —— 把"进行到哪"落到文件里，循环才能跨圈、跨会话连续（详见第 8 节）。`STOP` 文件默认不存在，需要急停时手动 `touch` 出来。

### 12.3 装到目标项目后会多出什么

`install-loop.sh` 跑完，目标项目会多出：上面 `.claude/` 整套 + `CLAUDE.md` + `.gitattributes` + `.mcp.json`(+`.example`)；`memory/` 里的运行态会被重置成空白。
- 老项目若已有 `CLAUDE.md` / `.mcp.json`，不会被覆盖——分别落成 `CLAUDE.loop.md` / `.mcp.loop.json` 等你手动合并。
- 你再加一个 `需求文档.md`（项目根），把客户需求写进去，就能开跑。
